项目名：IntelliTest 
团队名：ATCrafters

## 一、作业说明

### 1.1 作业概述

本作业要求开发一个 **AI 驱动的 AutoTestDesign 工具**，该工具能够执行软件需求分析、风险评估和系统化测试用例生成等功能。工具开发需符合 **ISTQB 基础级原则**以及 **ISO/IEC/IEEE 29119 标准**中定义的详细测试技术。

此外，需选定一个独立的目标应用（如 Web 应用、计算器、登录模块、待办事项列表等），使用开发的 AutoTestDesign 工具对该目标应用进行测试，以验证工具的有效性。

### 1.2 功能性需求（Functional Requirements）

| ID | 功能类别 | 描述 |
|---|---|---|
| FR 1.0 | 输入/解析 | 系统可从多种来源（CSV、纯文本、直接用户输入）导入软件需求 |
| FR 1.1 | 需求结构化 | 系统可解析和标记原始文本，识别关键组成部分：输入字段、数据范围、条件、预期动作 |
| FR 2.0 | 风险分析与优先级排序 | 系统可为每条导入的需求分配风险评分和测试优先级（High/Medium/Low） |
| FR 3.0 | 黑盒测试设计 | 系统可自动应用并生成至少三种 ISO 29119-4 核心黑盒技术（等价划分、边界值分析、决策表）的测试用例 |
| FR 4.0 | 白盒测试建模 | 系统可对系统行为建模（如状态转换图），并根据选定覆盖准则（如全状态覆盖）生成最优测试序列 |
| FR 5.0 | 测试预言生成 | 系统可针对给定需求和特定测试数据，合成预期结果 |
| FR 6.0 | 输出与导出 | 系统以结构化标准格式（JSON、Excel/CSV）生成测试工件（测试用例、测试套件、风险评分），可导入测试管理工具 |
| FR 7.0 | 测试套件优化 | 系统包含基于风险或覆盖效率对测试套件进行优先级排序或最小化的优化功能 |

### 1.3 非功能性需求

| 类别 | 要求 |
|---|---|
| 性能 | 测试用例生成时间不超过 2 秒；支持批量需求处理 |
| 可用性 | 提供直观的 Web UI；支持交互式审查与修改 |
| 安全性 | API Key 安全存储；输入数据隔离；防止注入攻击 |
| 可维护性 | 模块化架构；代码注释规范；配置文件管理 |

### 1.4 核心工作流

工具必须支持以下交互式设计审查流程：

```
概念定义 → 覆盖项识别 → 覆盖策略与方法选择 → 测试用例设计与可追溯性 →
提示设计 → 结果分析（测试用例与覆盖项/策略的映射） → 基于证据的改进
```

关键要求：**工具必须允许设计者（测试人员）在上述过程中交互式地审查、修订和更改设计项**，体现人的参与和有效性验证。

### 1.7 关键澄清

- 开发的是 **AutoTestDesign 工具本身**，然后用该工具测试一个**独立的目标应用**
- 风险分析报告、测试计划、详细测试设计文档**均针对目标应用**，而非工具本身
- 成本估算指使用工具测试目标应用的成本，可与手工测试对比
- 性能目标（≤2s）为强制性指标，若无法完全达成需在文档中提供详细分析和改进建议

### 2.3 核心模块技术实现路径

#### 2.3.1 输入解析引擎（FR 1.0）

- **输入格式支持**：CSV 文件上传（Pandas 解析）、纯文本粘贴、Web UI 表单直接输入
- **解析流程**：格式检测 → 字段映射 → 数据标准化 → 结构化存储
- **技术要点**：使用 Pandas 读取 CSV/Excel；文本输入通过正则预处理后送入 LLM 进行语义理解

#### 2.3.2 需求结构化引擎（FR 1.1）

- **核心思路**：利用 LLM 的 NLP 能力进行需求语句的语义解析
- **Prompt 设计策略**：
  - 定义结构化输出 Schema（JSON 格式）：`{input_fields, data_ranges, conditions, expected_actions}`
  - 使用 Few-shot Prompting 提供结构化示例
  - 输出验证：JSON Schema 校验 + 完整性检查
- **技术要点**：LangChain 的 StructuredOutputParser 或直接使用 Claude 的 Tool Use / Function Calling 实现可靠的结构化输出

#### 2.3.3 风险分析引擎（FR 2.0）

- **风险评分模型**：基于 ISO 31000 / ISTQB 风险分析原则
- **评分维度**：
  - 影响程度（Impact）：功能关键性、用户影响范围、数据敏感性
  - 发生概率（Likelihood）：需求复杂度、依赖数量、历史缺陷密度
- **计算公式**：`Risk Score = Impact × Likelihood`（1-5 量表，得分 1-25）
- **优先级映射**：High（15-25）、Medium（8-14）、Low（1-7）
- **技术要点**：LLM 对需求进行语义分析后给出各维度评分，结合规则引擎进行最终计算

#### 2.3.4 黑盒测试生成引擎（FR 3.0）

- **实现三种 ISO 29119-4 技术的自动化生成**：

| 技术 | 生成逻辑 | Prompt 策略 |
|---|---|---|
| **等价划分 (EP)** | 识别输入域 → 划分类别（有效/无效）→ 每个类别选代表值 | 要求 LLM 识别数据范围，按有效/无效类输出分区和代表值 |
| **边界值分析 (BVA)** | 基于 EP 结果 → 取边界点、边界±1 的值 | 对每个数值范围/长度约束，生成 min-1, min, min+1, max-1, max, max+1 |
| **决策表 (DT)** | 识别条件组合 → 生成决策矩阵 → 推导动作 | 提取条件项和动作项，生成完整的条件组合与对应预期结果 |

- **技术要点**：每种技术封装为独立的 Generator 类，统一实现 `generate(requirement) -> List[TestCase]` 接口；利用 LLM 进行语义级别的输入域识别和条件提取

#### 2.3.5 白盒测试建模引擎（FR 4.0）

- **建模流程**：
  1. LLM 分析需求文本，识别系统状态和状态转换条件
  2. 生成状态转换图（Mermaid 格式，前端渲染）
  3. 基于选定覆盖准则生成测试序列
- **覆盖准则支持**：全状态覆盖（All States）、全转换覆盖（All Transitions）
- **技术要点**：
  - 使用 LLM 将需求文字映射为 `(State, Event, Guard, Action, NextState)` 五元组
  - 图遍历算法（DFS/BFS）生成满足覆盖准则的最短路径序列
  - 前端使用 Mermaid.js 渲染状态图，支持交互式修改

#### 2.3.6 测试预言生成引擎（FR 5.0）

- **核心思路**：基于需求描述 + 具体测试输入，利用 LLM 推理预期输出
- **流程**：`需求描述 + 前置条件 + 测试输入数据 → LLM → 预期结果`
- **验证机制**：
  - 需求一致性校验（预期结果是否与需求逻辑一致）
  - 多轮推理（Chain-of-Thought）提升准确性
  - 人工审查确认（通过交互界面）
- **技术要点**：CoT Prompting 让 LLM 逐步推导；输出格式化为结构化预期结果

#### 2.3.7 输出与导出服务（FR 6.0）

- **支持格式**：
  - JSON：结构化测试用例完整数据（符合 ISO 29119-4 测试用例模板）
  - Excel/CSV：表格形式，含测试用例 ID、前置条件、测试步骤、预期结果、优先级等字段
- **技术要点**：openpyxl 生成 Excel（含格式样式）；JSON 按标准 Schema 输出，与 TestRail/Zephyr 等测试管理工具兼容

#### 2.3.8 测试套件优化引擎（FR 7.0）

- **优化策略**：
  - 基于风险的优先级排序：按 Risk Score 降序排列
  - 基于覆盖的最小化：去除冗余测试用例（覆盖相同路径的用例保留代表性样本）
  - 贪心覆盖最大化：在时间/资源约束下，选择覆盖最多未覆盖项的子集
- **技术要点**：
  - 覆盖矩阵构建（测试用例 × 覆盖项的二维矩阵）
  - 贪心算法 / 整数规划求解最优子集
  - 对比视图展示优化前后的覆盖率和用例数量变化

#### 2.3.9 交互式审查模块（核心交互要求）

- **设计理念**：每个 AI 生成的结果都需要经过"生成 → 人工审查 → 修改/确认 → 锁定"的工作流
- **交互功能**：
  - 覆盖项：增删改查，支持手动添加新的覆盖项
  - 覆盖策略：切换技术方法，调整参数
  - 测试用例：编辑字段、新增/删除用例、调整优先级
  - 所有修改记录可追溯（修改历史日志）
- **技术要点**：
  - 前端使用可编辑表格（Ant Design Editable Table）+ 内联编辑
  - 后端维护版本历史，支持撤销/重做
  - 修改标记（dirty flag）跟踪人工改动

### 2.4 数据流设计

```
用户输入需求 (CSV/Text/Form)
    │
    ▼

    │
    ▼
LLM 语义解析 ──────────► 结构化需求对象 (Input Fields, Conditions, etc.)
    │
    ├──────────────────────► 风险分析引擎 ─► 风险评分 & 优先级
    │
    ├──────────────────────► 黑盒测试生成引擎 ─► EP/BVA/DT 测试用例
    │
    ├──────────────────────► 白盒建模引擎 ─► 状态图 & 测试序列
    │
    ├──────────────────────► 测试预言引擎 ─► 预期结果
    │
    ▼
交互式审查 & 修改（人工参与）
    │
    ▼
测试套件优化 ───────────► 最终测试套件
    │
    ▼
导出 (JSON / Excel / CSV)
```

### 2.5 目标应用选择建议

建议选择以下类型之一作为测试目标：
- **Web 应用**：如登录注册系统（含表单验证、会话管理）
- **电商功能模块**：如购物车（含价格计算、库存校验）
- **API 服务**：如 RESTful CRUD 接口（含参数验证、错误处理）

选择原则：功能边界清晰、需求易于描述、覆盖项可量化，便于展示工具能力。

## 四、项目开发规范

### 4.1 代码规范

#### 4.1.1 Python（后端）

| 规范项 | 要求 |
|---|---|
| **代码风格** | 严格遵循 PEP 8；使用 Black 自动格式化，行宽 100 字符 |
| **类型注解** | 所有函数参数和返回值必须包含类型注解（PEP 484） |
| **文档字符串** | 公共 API 函数使用 Google style docstring，需包含 Args/Returns/Raises |
| **命名规范** | 类名：PascalCase；函数/变量：snake_case；常量：UPPER_SNAKE_CASE |
| **异常处理** | 业务异常使用自定义异常类；LLM 调用需有重试机制（max 3 retries） |
| **日志** | 使用 Python logging 模块；关键节点（LLM 调用、生成、导出）必须记录 |

#### 4.1.2 TypeScript / React（前端）

| 规范项 | 要求 |
|---|---|
| **代码风格** | 使用 ESLint + Prettier；遵循 Airbnb React 风格指南 |
| **类型检查** | 启用 strict 模式；禁止使用 `any` 类型（除非有明确注释说明） |
| **组件规范** | 每个组件一个文件；使用函数组件 + Hooks；避免 class 组件 |
| **命名规范** | 组件文件/名：PascalCase；工具函数：camelCase；常量：UPPER_SNAKE_CASE |
| **状态管理** | 全局状态使用 Zustand store；局部状态使用 useState/useReducer |

### 4.2 测试规范

#### 4.2.1 工具自身的测试

| 测试层级 | 要求 | 覆盖率目标 |
|---|---|---|
| **单元测试** | 每个 Generator 类、Parser 类、优化算法必须有对应单元测试 | ≥ 80% |
| **集成测试** | LLM 调用链（Prompt → 响应解析 → 结果验证）的集成测试 | 核心流程 100% |
| **E2E 测试** | 完整的"输入需求 → 生成测试用例 → 导出"流程测试 | 至少 1 条主路径 |

#### 4.2.2 目标应用的测试

- 使用 AutoTestDesign 工具为选定目标应用生成测试用例，然后使用 PyTest + Selenium（或对应框架）执行
- 测试用例必须覆盖多种黑盒技术和至少一种白盒技术
- 测试脚本需与工具生成的测试用例建立可追溯映射

### 4.3 版本控制规范

| 规范项 | 要求 |
|---|---|
| **分支策略** | `main`（稳定版本）+ `dev`（开发集成）+ `feature/<模块名>`（个人开发） |
| **Commit 规范** | 使用 Conventional Commits：`feat:` / `fix:` / `docs:` / `refactor:` / `test:` |
| **Code Review** | 合并到 dev 需至少 1 人 Review；合并到 main 需全组确认 |
| **提交频率** | 至少每日提交一次；禁止单次提交包含过多无关变更 |
| **.gitignore** | 排除 `__pycache__/`、`.env`、`node_modules/`、`*.pyc`、API Key 文件 |

### 4.4 AI / Prompt 工程规范

| 规范项 | 要求 |
|---|---|
| **Prompt 版本管理** | 所有 Prompt 模板存储为独立文件（`prompts/` 目录），纳入版本控制 |
| **Prompt 结构化** | 每个 Prompt 包含：System Prompt + User Prompt + Output Schema 三部分 |
| **可复现性** | LLM 调用记录输入/输出日志（开发阶段），便于调试和评估 |
| **Temperature 设置** | 结构化生成任务使用 temperature=0（确定性输出）；创意性任务 ≤ 0.3 |
| **降级策略** | LLM 调用失败时，返回明确的错误提示，不允许静默失败 |

### 4.5 测试行业标准对齐

本项目开发需对标以下行业标准与最佳实践：

| 标准/规范 | 对齐要求 |
|---|---|
| **ISO/IEC/IEEE 29119-4** | 黑盒测试技术（EP、BVA、DT）的实现需严格遵循该标准的定义和术语 |
| **ISTQB Foundation Level** | 测试用例结构（ID、前置条件、测试步骤、预期结果）、测试级别概念、风险分析原则需与 ISTQB 一致 |
| **ISO 31000** | 风险评分模型的风险识别、分析与评估维度对齐该风险管理标准 |
| **IEEE 829** | 测试计划文档结构可参照该标准（测试项、特性通过/失败准则、进度安排） |
