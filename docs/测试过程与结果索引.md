## 2.1 Skip 与 XFail 详细说明（给非项目成员）

### 先读懂 pytest 的四种「非正常通过」

| pytest 显示 | 通俗含义 | 是否说明被测软件坏了 |
| --- | --- | --- |
| **PASSED** | 实际结果符合测试预期 | 否（或符合「已知靶场行为」） |
| **FAILED** | 实际结果与预期不符，且**没预料到** | **是**，需要排查 |
| **SKIPPED** | **根本没测完**（条件不具备，主动跳过） | 不一定；常是环境/工具限制 |
| **XFAILED** | **测完了**，结果与「安全/质量期望」不符，但**我们事先知道会这样** | 对真实产品：是缺陷；对 Juice Shop：是故意漏洞 |

本次 **0 FAILED** 表示：没有「意料之外的自动化失败」。  
**3 SKIPPED** 表示：有 3 条用例因环境或方法限制未得到完整结论。  
**1 XFAILED** 表示：有 1 条用例**成功复现了已知安全问题**，脚本按设计标记为「预期失败」。

### XFail ×1：TC-SEC-014 / LR-014（SQL 注入登录绕过）

| 项目 | 说明 |
| --- | --- |
| **用例在测什么** | 在登录接口的「邮箱」字段输入 SQL 注入 payload，看系统是否仍错误地让用户登录。 |
| **测试数据** | 邮箱：`admin@juice-sh.op'--`；密码：任意（如 `anything`）。 |
| **正常产品的期望** | 应拒绝登录，返回 **HTTP 401**，**不签发** JWT 登录令牌。 |
| **Juice Shop 实际表现** | 返回 **HTTP 200**，响应里带有 `authentication.token`（JWT），相当于**无需正确密码即登录成功**。 |
| **为什么会这样** | OWASP Juice Shop 是**故意含漏洞的安全靶场**，登录接口把邮箱直接拼进 SQL 语句、未做参数化；`'` 与 `--` 可注释掉密码校验。这是官方设计的「SQL Injection」挑战，不是我们的测试脚本写错。 |
| **为何记为 XFail 而不是 Fail** | 脚本逻辑（`test_login_api.py` 131–136 行）：若发现 200+JWT，调用 `pytest.xfail(...)`，表示「从**安全验收**角度测试未通过，但在 Juice Shop 上**这是已知、可预期的观察结果**」。pytest 汇总为 **XFAIL**，**不计入 Failed**，也**不会把整次回归判红**。 |
| **缺陷编号** | **DEF-001**（Critical，故意漏洞） |
| **对答辩/文档怎么说** | 「我们验证了 LR-014：按安全标准应 Fail，在靶场上复现了 SQLi  bypass，测试自动化正确捕获并记录为 XFail。」 |
| **相关需求** | `login_requirements.csv` LR-014；设计见主文档 §7.6 TC-SEC-014 |

#### 这条用例是工具生成的还是人工加的？

**结论：需求与用例主体来自 IntelliTest 综合流程自动生成；预言、测试数据与可执行脚本经人工优化；不是从零手写的一条用例。**

| 阶段 | 来源 | LR-014 相关内容 | 文件定位 |
| --- | --- | --- | --- |
| **① 需求** | **人工编写** | 在 `login_requirements.csv` 中定义 LR-014，写明 Juice Shop 登录 SQL 未参数化、**可能 200 绕过**（故意漏洞场景） | `login_requirements.csv` 第 15 行 |
| **② 工具导出** | **IntelliTest 自动生成** | 综合设计（LLM 黑盒 EP/DT + 白盒 ST）导出 **3 条**候选用例：`LR-014_EP_002`、`LR-014_DT_001`、`tc-st-35016a48`；原始 oracle 多为「登录成功/绕过密码校验」等**漏洞观察**表述 | `未优化/LR-014_SQL injection tautology in email.csv` 全文 |
| **③ 套件优化** | 脚本清洗 | 补全部分占位数据；oracle 仍以工具原始观察为主 | `优化/LR-014_*.csv` |
| **④ 人工优化** | **人工审查/改写** | ① 将 oracle 改为「**安全期望 401；若 200+JWT 则记录 DEF-001**」（见 `02_手工优化说明.md` LR-014/015 行）；② 将 test_data 改为 Juice Shop 可执行 payload（如 `admin@juice-sh.op'--`）；③ **补漏 1 条** `SUP-LR-014-EP-003`（`' OR 1=1--` 第二种 payload，因需求 Data_Ranges 列两种而导出仅覆盖一种） | `人工优化/LR-014_*.csv`；说明见 `人工优化/02_手工优化说明.md` 第 68 行 |
| **⑤ 代表用例 TC-SEC-014** | **文档层归纳** | 从上述终稿中选取/映射为 1 条代表执行用例（对应 EP+Security），非另起炉灶新设计 | 主文档 §7.6 约 240 行 |
| **⑥ PyTest 自动化** | **人工编写脚本** | `test_lr014_sqli_tautology` 实现 TC-SEC-014；检测到 200+JWT 时 `pytest.xfail` | `scripts/target_app_tests/test_login_api.py` 131–136 行 |

**因此**：  
- **不是**测试同学完全手工凭空编造的用例——覆盖 LR-014 的 EP/DT/ST **骨架来自 AutoTestDesign 综合流程**；  
- **也不是**工具导出后原样执行——**安全验收标准（401 vs 200）与 Juice Shop 种子 payload 是人工优化写进去的**；  
- **XFail 逻辑**是执行阶段在 PyTest 里**人工实现**的，用于区分「靶场故意漏洞」与「自动化回归 Failed」。

### Skip ×1：TC-OAUTH-029 / LR-029（Google OAuth 登录跳转）

| 项目 | 说明 |
| --- | --- |
| **用例在测什么** | 打开登录页 `/#/login`，点击 **「使用 Google 登录」** 按钮，浏览器应跳转到 `accounts.google.com` 的 OAuth 授权页。 |
| **测试方式** | Playwright 打开真实浏览器访问 localhost:3001（`test_login_ui.py` 142–150 行）。 |
| **实际发生了什么** | 页面上**找不到任何 Google 登录按钮**（按钮、链接、Google 图标均为 0 个）。脚本无法执行「点击并跳转」这一步。 |
| **为什么会这样** | Juice Shop 仅在当前访问地址（origin）落在配置项 `googleOauth.authorizedRedirects` 白名单内时，才显示 Google 登录。本地 `http://localhost:3001` 在本部署中 **OAuth 不可用**（`oauthUnavailable=true`），按钮被前端隐藏。 |
| **为何记为 Skip 而不是 Fail** | 不是「点了按钮但没跳转」，而是**前置条件不满足**（按钮不存在）。Skip 表示「本环境测不了正向 OAuth 流程」，**不代表 OAuth 功能一定坏了**。 |
| **与 TC-OAUTH-030 的关系** | LR-030（**Pass**）测的是「未授权来源下 OAuth 按钮应隐藏」——在 localhost 上**正好观察到按钮隐藏**，与 LR-030 一致；LR-029 需要的则是**另一个**「已授权、按钮可见」的环境。 |
| **怎样才能测完 LR-029** | 在 `authorizedRedirects` 包含当前 origin 的部署上重跑；或改 hosts/反向代理使 origin 与白名单一致。 |
| **对答辩/文档怎么说** | 「LR-029 需 Google 授权环境；本地 Docker 未展示 OAuth 按钮，故 Skip。LR-030 在同环境 Pass，说明隐藏逻辑符合预期。」 |

### Skip ×1：TC-REL-019 / LR-019（忘记密码接口速率限制）

| 项目 | 说明 |
| --- | --- |
| **用例在测什么** | 对照 LR-018（登录接口无限速）：验证 **「忘记密码」** 接口 `POST /rest/user/reset-password` 是否有频率限制——同一 IP 在 5 分钟内超过约 100 次后，第 101 次应返回 **HTTP 429 Too Many Requests**。 |
| **测试方式** | 用脚本连续发送 15 次 reset-password 请求（`test_login_api.py` 192–198 行；完整用例设计为 101 次）。 |
| **实际发生了什么** | **每一次**请求都返回 **HTTP 500**（服务器内部错误），**从未出现 429**。 |
| **为什么会这样** | 本地 Docker 通常**未配置邮件服务**（SMTP）；Juice Shop 在处理 reset-password 时可能因发信失败或**反自动化/安全中间件**直接返回 500，请求在到达「计数并返回 429」逻辑之前就失败了。 |
| **为何记为 Skip 而不是 Fail** | 我们**无法区分**「限速逻辑坏了」还是「环境根本走不到限速分支」。在 500 占满响应时，继续断言 429 没有依据，故 Skip。 |
| **设计意图（背景）** | LR-019 是**对照用例**：Juice Shop 故意让**登录**不限速（LR-018 已 Pass 并记录 RISK-001），而 **reset-password 有限速**；本用例要证明「至少有一个认证相关接口做了限速」。本地未完成后半段验证。 |
| **怎样才能测完 LR-019** | 配置可用邮件服务或使用官方推荐的 staging 环境；或在测试环境 mock 邮件发送，使接口能正常进入 rate-limit 中间件。 |
| **对答辩/文档怎么说** | 「忘记密码接口的 429 阈值在 localhost 未能验证（全程 500），标记 Skip；登录无限速已在 LR-018 证实。」 |

### Skip ×1：TC-ERR-022 / LR-022（后端 500 错误时前端如何处理）

| 项目 | 说明 |
| --- | --- |
| **用例在测什么** | 当用户点击登录时，若**后端数据库查询失败**（例如 DB 宕机），服务器应返回 **HTTP 500**；浏览器端应清除 token、清除 bid、显示错误，且不应留下「半登录」状态。 |
| **测试方式** | 设计为**故障注入**：在登录请求发出前，人为让数据库不可用或让 Sequelize 抛错，再观察前端与 API 行为。 |
| **实际发生了什么** | 自动化套件**没有执行**上述故障注入；脚本入口直接 `pytest.skip(...)`（`test_login_api.py` 326–327 行）。 |
| **为什么会这样** | 在共享的 Docker 开发环境中**主动弄挂数据库**风险高（会影响其他测试、需人工恢复）。当前 PyTest 套件只覆盖「API 正常/异常输入路径」，**不包含**运维级混沌测试。 |
| **为何记为 Skip 而不是 Fail** | 用例**未执行**，不是「执行了但前端处理错了」。属于 **Blocked（环境/方法未就绪）**，不是功能回归失败。 |
| **怎样才能测完 LR-022** | 单独开测试容器：短暂 `docker pause` 数据库容器、使用 mock/stub 强制 login 路由返回 500，或用手动/Playwright 在受控环境触发。 |
| **对答辩/文档怎么说** | 「LR-022 需 DB 故障注入，未纳入本次自动化回归，标记 Skip；属测试方法限制，非登录成功路径缺陷。」 |
