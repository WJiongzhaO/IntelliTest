# 测试套件与审查结果持久化改进建议

## 背景

当前 IntelliTest 已经支持需求录入、需求结构化、风险分析、黑盒测试生成、白盒建模、测试预言和审查导出等流程。实际测试时发现：Docker 或后端服务重启后，需求列表仍然存在，但“审查导出”页面中的测试用例、覆盖项和审查结果会清空。

该现象说明当前系统只对需求及其结构化/风险信息做了数据库持久化，而测试套件相关数据仍然主要依赖内存或前端会话。

## 当前表现

当前系统的数据生命周期如下：

- 需求标题、需求原文、结构化字段、风险评分会写入后端 SQLite 数据库，服务重启后仍可恢复。
- 黑盒测试用例、综合测试套件、白盒状态模型、测试预言结果、覆盖项和审查修改历史没有完整写入数据库。
- 审查导出页面读取的是当前浏览器会话中最近一次生成或保存的测试套件。
- 如果用户生成 A 需求的测试用例后未导出，又继续生成 B 需求的测试用例，审查导出页面会以最新测试套件为准，旧结果可能被覆盖。
- Docker 容器或后端进程重启后，后端内存中的测试套件会丢失；若前端会话数据也不可用，则审查导出页面会显示暂无待导出用例。

补充：需求自身已支持 `title` 字段并写入后端 SQLite 数据库。该字段属于需求基础数据，和 `raw_text`、结构化字段、风险评分一起跨服务重启保留；当前尚未持久化的是测试套件、覆盖项、测试预言和审查历史等测试设计结果。

## 影响

该限制不会影响当前 FR 6.0 的“对当前测试工件进行审查并导出 JSON/CSV/Excel”的核心演示流程，但会影响以下长期使用场景：

- 无法跨服务重启恢复历史测试结果。
- 无法按需求查看过去生成的测试套件。
- 多名组员或多台电脑无法共享同一批审查后的测试用例。
- 无法形成完整的审查历史和可追溯审计记录。
- 用户需要在生成一批测试结果后及时导出，否则后续生成结果可能覆盖当前待导出内容。

## 改进目标

建议后续版本增加测试套件与审查结果的后端持久化能力，使系统支持：

- 按需求保存多次生成的测试套件。
- 按测试套件恢复测试用例、覆盖项和预言结果。
- 保存人工审查修改历史。
- 支持跨浏览器、跨设备和后端重启后的结果恢复。
- 导出页面可从后端选择历史测试套件，而不是只依赖当前会话。

## 建议的数据模型

可在后端数据库中增加以下表或等价模型：

- `test_suites`：保存测试套件基本信息，如套件编号、关联需求、生成技术、创建时间。
- `test_cases`：保存测试用例标题、前置条件、步骤、测试数据、预期结果、优先级、风险分数、技术类型。
- `coverage_items`：保存覆盖项编号、描述、类型、关联需求和关联测试用例。
- `oracle_results`：保存测试预言、推理步骤、一致性检查结果和审查状态。
- `review_history`：保存人工修改记录，包括修改对象、字段、修改前后内容、修改时间。

## 建议的接口设计

可增加以下 API：

- `POST /api/v1/suites`：保存新生成的测试套件。
- `GET /api/v1/suites?requirement_id=...`：查询某个需求下的历史测试套件。
- `GET /api/v1/suites/{suite_id}`：读取测试套件详情。
- `PUT /api/v1/suites/{suite_id}/review`：保存审查后的测试用例、覆盖项和修改历史。
- `DELETE /api/v1/suites/{suite_id}`：删除不需要的测试套件。
- `POST /api/v1/export/artifact?suite_id=...`：直接基于后端持久化的套件导出。

## 前端改进建议

审查导出页面可增加以下交互：

- 在页面顶部增加“选择测试套件”下拉框。
- 显示测试套件所属需求、生成时间、测试技术和用例数量。
- 支持“保存审查到后端”按钮。
- 支持“另存为新版本”，避免覆盖旧审查结果。
- 当当前套件只存在于浏览器会话中时，显示“尚未持久化”的提示。

## 与成员 E 当前工作的关系

成员 E 当前实现的 FR 6.0 导出功能负责对当前测试工件进行交互式审查，并导出 JSON、CSV、Excel 文件。该功能已经满足课程演示中“生成后审查并导出”的核心要求。

本改进建议属于后续增强方向，涉及后端数据库模型、测试套件管理接口和跨会话恢复机制，范围已经超出单纯的导出格式实现。当前阶段建议在文档中明确限制，并在实际使用时遵循“生成一批测试结果后及时导出”的操作流程。

## 当前使用建议

在未实现后端持久化前，建议用户按以下方式使用：

1. 完成需求结构化和风险分析。
2. 生成某个需求的测试用例或综合测试套件。
3. 立即进入审查导出页面检查并导出 JSON/CSV/Excel。
4. 确认导出文件无误后，再生成下一批需求的测试结果。

这样可以避免后续生成操作覆盖当前待导出的测试套件。
