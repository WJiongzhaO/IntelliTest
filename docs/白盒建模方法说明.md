# 白盒建模方法说明（FR 4.0）

## 1. 适用范围

IntelliTest 中 FR 4.0 采用 **ISTQB 状态测试** 思路：从结构化需求抽取系统行为的状态机，再按覆盖准则生成测试序列。  
本模块面向 **规格/行为白盒**（状态与转换），不解析被测程序源代码；与 FR 3.0 黑盒（EP/BVA/DT）互补。

## 2. 五元组定义

每条转换表示为：

| 字段 | 含义 | 示例 |
|------|------|------|
| `state` | 当前状态 | `LoggedOut` |
| `event` | 触发事件/输入 | `login` |
| `guard` | 守卫条件（可选） | `valid credentials` |
| `action` | 迁移动作（可选） | `authenticate` |
| `next_state` | 目标状态 | `Active` |

转换标识（用于覆盖追踪）：`LoggedOut--login-->Active`

## 3. 覆盖准则对比

| 准则 | 目标 | 典型策略 | 适用场景 |
|------|------|----------|----------|
| **ALL_STATES** | 每个状态至少访问一次 | 从初始状态贪心扩展 + BFS 最短路补未访问状态 | 状态数较少、关注生命周期 |
| **ALL_TRANSITIONS** | 每条转换至少触发一次 | 逐条覆盖未触发边，必要时 BFS 到达边起点 | 转换逻辑复杂、回归风险在事件组合 |

算法实现：`coverage_planner.py`（BFS 最短路 + 贪心覆盖，非穷举 DFS 以避免路径爆炸）。

## 4. Mermaid 状态图

- LLM 或 `mermaid_renderer.py` 生成 `stateDiagram-v2` 语法。
- 前端 `WhiteboxWorkbench` 渲染 SVG，支持人工编辑 Mermaid 后 `PUT /whitebox/model/{id}` 重算序列。

## 5. 与黑盒/风险/需求的衔接

1. **成员 A**：`POST /requirements/{id}/structure` 写入 DB 字段。  
2. **成员 B**：`POST /requirements/{id}/risk` 写入 `risk_score` / `priority`。  
3. **成员 D**：通过 `requirement_id` 从 DB 加载为 `StructuredRequirement`，映射到 `TestCase.priority` / `risk_score`。  
4. **成员 C**：`POST /test-design/combined` 中 `techniques` 含 `EP`/`BVA`/`DT` 时自动合并黑盒用例。  
5. **成员 D FR 5.0**：对缺 `expected_result` 的用例批量 CoT 合成预言。

## 6. API 摘要

- `POST /api/v1/whitebox/model` — 建模并生成序列与用例  
- `PUT /api/v1/whitebox/model/{id}` — 人工修订后重算  
- `POST /api/v1/test-design/combined` — 黑盒 + 白盒 + 预言流水线  

完整契约见 Swagger：`http://localhost:8000/docs`。
