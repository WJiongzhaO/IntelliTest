# 详细测试设计与执行文档（成员 D 主导）

## 1. 被测需求示例

**需求 ID**：`req-login-demo`（可在 Requirements 页录入后 Structure + Risk）

**原文**：用户使用有效凭据登录后进入活动会话；用户登出后会话结束。

**结构化字段**（示例）：

- 输入：`username`, `password`
- 条件：`credentials valid`, `user clicks logout`
- 期望动作：`authenticate user`, `show dashboard`, `clear session`

## 2. 白盒用例设计表（StateTransition）

| 用例 ID | 覆盖项 | 状态路径 | 事件序列 | 优先级 |
|---------|--------|----------|----------|--------|
| tc-st-001 | LoggedOut, Active | LoggedOut → Active | login | 由 risk_score 推导 |
| tc-st-002 | Active--logout-->LoggedOut | Active → LoggedOut | logout | 同上 |

生成方式：`POST /api/v1/whitebox/model`，`coverage=ALL_TRANSITIONS`。

## 3. 黑盒用例（成员 C，合并示例）

| 用例 ID | 技术 | 标题摘要 |
|---------|------|----------|
| req-login-demo_EP_001 | EP | 有效等价类代表值 |
| req-login-demo_BVA_001 | BVA | 边界长度/范围 |

生成方式：`POST /api/v1/test-design/combined`，`techniques=["EP","BVA","DT","StateTransition"]`。

## 4. 测试预言样例（FR 5.0）

**用例**：tc-st-001  

**CoT 推理链（示例）**：

1. 解析 test_data 中的有效凭据。  
2. 应用条件 credentials valid。  
3. 对照 expected_actions：authenticate、show dashboard。  
4. 结论：用户认证成功并展示仪表盘。

**预期结果**：`User is authenticated and the dashboard is displayed.`

审查：`PUT /api/v1/oracle/{oracle_id}/review`，`action=confirm`，`sync_test_case=true` 回写用例。

## 7. 端到端操作步骤

1. Requirements：录入 → Structure → Risk  
2. Test Design Workbench：选择需求 → 勾选技术 → 生成套件  
3. Oracle Editor：选择用例 → 合成/确认预言  
4. 导出：后续对接 FR 6.0 export（若已实现）或复制 Swagger 响应 JSON
