# 成员 E - 测试计划：成本估算、测试框架选择及理由

## 1. 文档目的

本文档对应课程期末项目交付物中的「测试计划」部分，聚焦成员 E 负责的内容：

- 成本估算
- 测试框架选择及理由
- 与团队组织和进度安排相关的执行说明

根据项目规划文档约定，所有测试计划内容均针对**自选目标应用**的测试活动，而不是测试 IntelliTest 工具本身。IntelliTest 是 AutoTestDesign 工具，用于辅助测试人员生成、审查、优化并导出目标应用的测试工件。

## 2. 测试对象与范围假设

本成本估算以一个典型小型 Web 目标应用为估算对象，例如登录/注册模块、待办事项列表或简单 CRUD 系统。该类目标应用通常包含：

- 表单输入与字段校验
- 正常业务路径
- 异常输入处理
- 状态变化或权限变化
- 基础非功能要求，如易用性、响应时间和安全输入校验

测试范围包括：

- 需求导入与结构化后的人工确认
- 风险评分与优先级确认
- 黑盒测试用例设计，包括等价类划分、边界值分析、决策表
- 至少一种白盒或状态转换视角的测试设计
- 测试预言审查，即预期结果确认
- 测试用例导出为 JSON、CSV 或 Excel
- 使用选定测试框架在目标应用上执行核心测试脚本

## 3. 测试框架选择

### 3.1 推荐框架组合

| 测试层级 | 推荐框架 | 使用目的 |
|---|---|---|
| Web E2E 测试 | Selenium + PyTest | 执行浏览器级端到端测试，验证目标应用真实交互 |
| API 测试 | PyTest + Requests | 验证目标应用后端接口的输入、输出、状态码和错误处理 |
| 测试数据管理 | PyTest fixture | 管理登录账号、初始数据、清理流程和重复测试数据 |
| 测试报告 | pytest-html 或 Allure（可选） | 生成可展示的执行报告，辅助最终文档与答辩 |
| 测试工件导出 | IntelliTest FR 6.0 Export Service | 输出 JSON/CSV/Excel 测试用例和覆盖追溯数据 |

### 3.2 选择 PyTest 的理由

PyTest 适合作为目标应用测试执行框架，理由如下：

- 与项目后端 Python 技术栈一致，团队学习和维护成本较低。
- fixture 机制适合组织登录态、测试数据和环境初始化。
- 参数化能力强，可以直接消费 IntelliTest 导出的 CSV/JSON 测试数据。
- 插件生态成熟，可扩展覆盖率、HTML 报告、并发执行和失败重试。
- 测试脚本结构清晰，便于在详细测试设计与执行文档中展示。

### 3.3 选择 Selenium 的理由

若目标应用为 Web 应用，Selenium 适合作为浏览器自动化框架，理由如下：

- 能验证真实用户操作路径，例如输入、点击、跳转、错误提示和页面状态变化。
- 支持 Chrome、Edge 等主流浏览器，便于课程演示。
- 与 PyTest 结合后，可以将 IntelliTest 生成的测试步骤映射为自动化脚本。
- 对登录、表单校验、CRUD 等典型目标应用场景覆盖充分。

### 3.4 选择 Requests 的理由

若目标应用包含 REST API，Requests 适合做接口级测试，理由如下：

- 调用方式简单，适合快速验证参数校验、业务状态码和响应体。
- 相比浏览器测试运行更快，适合补充边界值和决策表产生的大量用例。
- 可以与 PyTest 参数化结合，直接批量执行 IntelliTest 导出的测试数据。

## 4. IntelliTest 与测试框架的协作方式

IntelliTest 不直接替代测试框架，而是承担测试设计和工件生成角色。推荐协作流程如下：

1. 在 IntelliTest 前端导入目标应用需求。
2. 使用成员 A 模块完成需求结构化。
3. 使用成员 B 模块生成风险评分与优先级。
4. 使用成员 C/D 模块生成黑盒和白盒测试用例。
5. 使用成员 E 的交互式审查页面人工修改测试用例、覆盖项和预期结果。
6. 通过 FR 6.0 导出 JSON、CSV 或 Excel。
7. 使用 PyTest/Selenium/Requests 读取导出的工件，执行目标应用测试。
8. 将执行结果回填到详细测试设计与执行文档中。

该流程能体现作业要求中的「设计者参与」和「交互式验证有效性」。

## 5. 成本估算方法

成本估算采用人时估算方式，并与纯手工测试进行对比。估算维度包括：

- 需求理解与梳理时间
- 测试用例设计时间
- 风险分析时间
- 预期结果确认时间
- 自动化脚本编写时间
- 测试执行与结果整理时间
- 测试工件导出与报告整理时间

## 8. 成本对比与收益分析

| 对比项 | 使用 IntelliTest | 纯手工测试 | 差异 |
|---|---:|---:|---|
| 总人时 | 12.8 | 18.5 | 节省约 30.8% |
| 测试设计一致性 | 高 | 中 | 工具统一输出字段和覆盖追溯 |
| 覆盖项追踪 | 自动生成后人工确认 | 依赖人工维护 | 工具更稳定 |
| 导出工件 | 自动 JSON/CSV/Excel | 人工整理 | 工具节省报告整理时间 |
| 可复用性 | 高 | 中 | Prompt、测试数据和脚本可复用 |
| 初始学习成本 | 中 | 低 | 工具首次使用需要熟悉流程 |

使用 IntelliTest 的主要收益不只是减少人时，还包括：

- 测试用例结构更统一。
- 覆盖项、测试策略和测试用例之间更容易追溯。
- 风险优先级能辅助测试执行排序。
- 导出工件更适合放入交付材料和测试管理工具。
- 人工审查记录能体现测试设计者参与，符合课程要求。

## 9. 风险与限制

| 风险 | 影响 | 应对措施 |
|---|---|---|
| LLM 输出不稳定 | 可能导致结构化字段或预期结果不准确 | 使用人工审查页面确认并修改 |
| 生成性能不稳定 | 可能无法稳定满足 2 秒目标 | 文档中记录环境、耗时和改进建议 |
| 目标应用需求描述不清 | 测试用例质量下降 | 先补充需求描述和输入约束 |
| 自动化脚本与导出字段不匹配 | 执行成本上升 | 固定导出字段，并在脚本中统一解析 |
| 团队接口变更 | 前端对接失败 | 以 dev 分支为集成基线，合并前统一运行接口检查 |

## 10. 推荐进度安排

| 阶段 | 工作内容 | 负责人 |
|---|---|---|
| 第 1 阶段 | 确认目标应用与需求文本 | 全组 |
| 第 2 阶段 | 导入需求、结构化、风险分析 | 成员 A/B + 成员 E 前端支持 |
| 第 3 阶段 | 生成黑盒/白盒测试用例与测试预言 | 成员 C/D |
| 第 4 阶段 | 交互式审查、修改、导出测试工件 | 成员 E |
| 第 5 阶段 | 使用 PyTest/Selenium/Requests 执行测试 | 测试执行者 |
| 第 6 阶段 | 汇总成本估算、测试结果和演示材料 | 全组 |

## 11. 结论

本项目建议采用 PyTest 作为统一测试执行框架，并根据目标应用类型选择 Selenium 或 Requests。IntelliTest 负责测试设计、风险分析、交互式审查和工件导出，测试框架负责执行目标应用测试。

在小型目标应用场景下，使用 IntelliTest 预计可将测试活动从约 18.5 人时降低到约 12.8 人时，同时提升测试工件规范性、可追溯性和演示完整度。
